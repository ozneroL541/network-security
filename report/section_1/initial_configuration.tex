\section{Greenbone}
\label{sec:greenbone}
Greenbone/OpenVAS is an open-source vulnerability 
scanning tool that provides comprehensive security 
assessments for IT systems.
To set up Greenbone/OpenVAS, firstly download the official 
Greenbone/OpenVAS OVA file from the Greenbone website.\\
\href{https://www.greenbone.net/en/openvas-free/}{https://www.greenbone.net/en/openvas-free/} \\
Then, import the OVA file into
VirtualBox 6.1 or higher.
Assign at least 5GB of RAM and 2 CPU cores to the virtual machine. 
It's better to use about 12GB of RAM.
For the first configuration, use the bridge network adapter, which allows the virtual machine to be on the same network as the host machine.
Start the Greenbone/OpenVAS virtual machine and follow the on-screen instructions to complete the initial setup.
Assign a user for the web interface.
Once the setup is complete, access the Greenbone/OpenVAS update the feed 
by going to \texttt{Maintenance>Feed>Update} and wait it to synchronize
(ususally hours).

\subsection{Docker Vulnet}
\label{sec:docker-vulnet}
To simulate a complex network environment, we are using a 
docker container with multiple vulnerable services.
The scope of this network is to show the capabilities 
of Greenbone/OpenVAS in scanning a complex system.
\begin{figure}[h!]
    \label{fig:docker-vulnet-conf}
    \centering
    \includegraphics[width=0.8\textwidth]{../project/docker-vulnet/container-network.png}
    \caption{Docker network configuration}
\end{figure}
To set up the network we used the following docker compose file:
\inputminted{yaml}{../project/docker-vulnet/docker-compose.yml}

To ensure proper network setup it was crucial to set the correct
routes throgh all the containers, so that they could communicate 
with each other in the way we wanted. 
To do so we wrote the following script:
\inputminted{bash}{../project/docker-vulnet/setup.sh}

To let Greenbone/OpenVAS scan the containers, 
we had to crete further rules.
Since the VM is inteded as a vulnerable machine,
we had no intention to let the firewall block any traffic, 
therefore, firstly, we flushed all its rules and then 
we build up our own ruleset, which allows all the traffic 
from the Greenbone/OpenVAS to the container's network.
\inputminted{bash}{../project/docker-vulnet/hardcore-setup.sh}

\subsubsection{vulnbox1}
\label{sec:vulnbox1}
The first vulnbox was intentended to be vulnerable to 
log4Shell (CVE-2021-44228), a critical vulnerability 
in the Apache Log4j library.
However, since the container was handmade, 
Greenbone/OpenVAS was not able to detect it.
\inputminted{yaml}{../project/docker-vulnet/network/vulnbox-1/Dockerfile}
Even it the vulnerability was not detected, 
the main role of the first vulnbox was to route all the traffic 
to the docker network and vice versa and, in that sense, 
it was successful.
