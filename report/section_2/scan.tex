\section{OpenVAS Scan}
In this section it will be explained how to start a scan (that could be just for a host, or a group of hosts). We will create a target, which means declaring which host (or group of hosts) to be scanned, which port range and protocol (TCP or UDP).

This is the scan window, where you will create the tasks which are configurations for the targets and ports that are going to be scanned, how many NVTs (Network Vulnerability Test) are going to be used for the scan (done in the Scan config windows under Configuration), how many concurrent hosts are being scanned at a given moment etc. We will get to all of the configuration in time.

\subsection{Configuration $\rightarrow$ Ports}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.9\textwidth]{../presentation/images/openvas_ports_interface1.png}
    \caption{Port creation interface}
\end{figure}




This is the ports creation interface, you can create custom ports ranges looking for services that use TCP/UDP. For the most part OpenVAS works by giving us templates, in this case, they are for a complete scan of all IANA (Internet Assigned Numbers Authority) ports, which means it will scan for all $65536$ ports only TCP, UDP or both. It really just depends on which preset is chosen. 

OpenVAS encodes ports in a format like \textbf{T:$1-1023$, $4500$,U:$2-100$, $3246$} which means that it will scan for TCP ports between $1$ and $1023$ and $4500$ alone, and for UDP ports from $2$ to $100$ and $3246$ alone. In this way you can define ports range and choose to do only TCP or UDP scans indicated by \textbf{T:} or \textbf{U:}.

Under the hood OpenVAS uses Nmap to scan the ports, and just like it UDP ports scan will take considerably longer than TCP. This is because when a TCP request is made, the server either accept the connection or it rejects it, while for UDP being a connectionless protocol will wait until timeout. So beware about this aspect. 

For our experiments we will create two presets: one for ftp service (port $21$) and one for an http server (port $80$). Go on the add port icon on the top of the interface (the little square with a plus), and you should get this popup.

\subsubsection{Credentials for target access}
\begin{figure}[H]

Credentials are used for deeper scans, you give the credentials to OpenVAS and it will try to login to the machine, scanning the host from within, looking at configuration files, installed packages. This is done to improve scan performance since it flags problems that are not visible from the outside, but may lead to exploitable vulnerabilities. This function can also be used to check for default credentials if they are in use or no.

This task will scan Windows XP for the FTP service. The options are pretty much self explanatory since you are going to use all the presets set before, the important flag to enable is the alterable task flag just for our convenience. This flag permits you to modify the task at will, otherwise if set to \textbf{no} it would not permit you to alter the task once created, and thus you will have to redo everything. This flag is normally set to false because when you scan a system for vulnerabilities or problems, you would want to correct them, then to verify if the problems have been mitigated you would simply clone the task that you previously executed, start it without modifications (if the alterable flag was true then you would be able to modify it) and then check if all problems are fixed. If you still have problems to be fixed then you should repeat until there are no more.
 
Now you should have a new task created with the name you gave it, in this case \textbf{WIN XP FTP TASK} like so:

Reports are the result of the scan activity, there you will find all the information regarding the scan done, such as host, open ports, vulnerabilities found with its severity score:
